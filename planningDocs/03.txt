## 2. Models used (final, realistic)

### A. Socratic Tutor Model (pedagogy first)

**eth-nlped/TutorRL-7B**

* Fine-tuned with RL for *tutoring behavior*
* Trained to:

  * ask questions
  * scaffold reasoning
  * avoid giving solutions
* Domain: math → **behavior transfers to code**

👉 Role: **HOW to talk**, not WHAT to fix.

---

### B. Optional lightweight Socratic model

**eurecom-ds/phi-3-mini-4k-socratic**

* Smaller, cheaper
* Very strict Socratic style
* Used when:

  * low latency
  * cheap hints
  * early-stage guidance

---

### C. Code Reasoner Model (logic first)

One of:

* **CodeLlama-Instruct**
* **WizardCoder-Instruct**
* **Qwen-Coder**
* or a strong hosted general model

👉 Role: **understand the bug**, not talk to the student.

---

### D. Verifier Model (judge, not generator)

Can be:

* same model as code reasoner
* or cheaper general LLM

👉 Role: **reject bad hints**, score pedagogy + correctness.

---

## 3. Why we STILL don’t rely on a single model

Even TutorRL-7B **cannot**:

* deeply analyze complex code
* reason about edge cases reliably
* scale across languages alone

So:

> **Pedagogy ≠ reasoning ≠ verification**

They are **separate responsibilities**.

This is the key architectural insight.

## 4. The Socratic Brain (step by step)

### Step 1: Input arrives

From frontend:

* buggy code
* selected language
* runtime error (if any)
* history of hints used
* time elapsed

---

### Step 2: RAG (logic grounding, API-only)

**Goal:** help reasoning models, especially SLMs.

* Code corpus:

  * buggy + fixed examples
  * algorithms
  * common mistakes
* Vector DB:

  * hosted (Pinecone / Weaviate / hosted FAISS)
* Embeddings:

  * hosted embedding APIs

RAG returns:

* *conceptual references*, not copy-paste code

👉 Used ONLY internally, never shown to user.

---

### Step 3: Planner (TreeInstruct-style, shallow)

Planner decides **intent**, not text.

Output example:

```json
{
  "bug_class": "off-by-one",
  "hint_level": 2,
  "strategy": "question",
  "target_concept": "loop boundary"
}
```

No generation yet.

---

### Step 4: Code Reasoner (logic extraction)

Prompted with:

* code
* error
* RAG snippets
* planner intent

Output:

* internal analysis
* *what is wrong*
* *where to guide the student*

This output is **not user-facing**.

---

### Step 5: Socratic Tutor (TutorRL-7B)

TutorRL-7B receives:

* planner intent
* reasoning summary (not full solution)
* strict rules:

  * ask ONE question
  * no fixes
  * no code dumps

It generates **multiple candidate hints**.

---

### Step 6: Verifier (EduDial enforcement)

Verifier scores each hint on:

* avoids solution
* targets correct bug
* Socratic tone
* progression vs repetition

Only the **best hint survives**.

This is where:

> architecture replaces fine-tuning.

---

### Step 7: Hint returned to frontend

Frontend shows:

* hint text
* penalty counter (+1)
* time keeps running

## 6. Game logic (simple, measurable)

Winner decided by:

```
score = 
  correctness_weight
- (hint_count × hint_penalty)
- (time_taken × time_penalty)
```

Displayed:

* time
* hints used
* solved / not solved

That’s it. No complexity creep.

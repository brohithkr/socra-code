## 1. Backend stack (recommended, minimal, correct)

### **Primary backend**

**Python + FastAPI**

This is the backbone.

### Why FastAPI (and not Express / Flask)?

* Native async → perfect for **LLM calls + WebSockets**
* Easy background tasks (benchmarks, scoring)
* Python ecosystem → **RAG, embeddings, code execution**
* Clean OpenAPI → frontend integration is trivial
* Same language as:

  * LLM orchestration
  * RAG
  * benchmarking scripts

👉 This avoids glue code across Node + Python.

## 3. Backend technologies (exact list)

### 3.1 API layer

**FastAPI**

* REST endpoints
* WebSocket endpoints (multiplayer)
* JSON-first
* Async

Used for:

* Create game session
* Join game via code
* Run code
* Request hint
* Submit solution

---

### 3.2 Real-time multiplayer

**FastAPI WebSockets**

Why WebSockets?

* Shared editor state
* Timer sync
* Hint penalties broadcast
* Game score updates

Each game room:

* One WebSocket channel
* Multiple clients
* Server is authoritative

No need for Socket.IO unless you want reconnection sugar.

---

### 3.3 Game & session state

**Redis (recommended)**
Fallback: in-memory dict (dev only)

Redis stores:

* room_id → players
* code snapshot
* hint count
* time started
* scores

Why Redis?

* Fast
* Shared state
* TTL for rooms
* Simple key-value

Example:

```
game:{room_id}:state
game:{room_id}:code
game:{room_id}:scores
```

## 4. Socratic backend (core intelligence)

This is **pure Python**, no framework coupling.

### 4.1 Planner (TreeInstruct-style)

`planner.py`

* Input:

  * buggy code
  * error output
  * past hints
* Output:

  * target concept (e.g. “loop boundary”)
  * hint intent (question / nudge / clarification)

Implementation:

* Prompt-based reasoning
* No tree search explosion
* Depth-1 or depth-2 planning only

---

### 4.2 Generator (LLM)

`generator.py`

* Calls:

  * OpenRouter / Groq / local model
* Generates:

  * 3–5 Socratic hint candidates

Models:

* SOTA: Llama 3.3 70B, GPT-OSS, Gemini Flash
* SLM: Llama 3 8B, Mistral 7B, Qwen 7B

---

### 4.3 Verifier (no training)

`verifier.py`

This is **NOT a model you train**.

It is:

> a **judge LLM call**

Input:

* buggy code
* hint candidate
* planning intent

Output:

* numeric score (0–10)
* optional reason

You select the **highest scoring hint**.

This replaces RLHF and fine-tuning.

---

### 4.4 RAG engine

`rag.py`

**Technologies used:**

* FAISS (vector search)
* Sentence-transformers / code embeddings
* Plain filesystem for code corpus

Corpus:

* Python / Java / C++ files
* Known buggy + correct versions
* Treated as *logic reference*, not syntax oracle

RAG flow:

```
query = embed(buggy_code + error)
snippets = faiss.search(query)
inject snippets into generator prompt
```

This is **critical for SLMs**.
